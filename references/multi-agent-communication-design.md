# Agent Creator - 多 Agent 通信设计项（v1）

> 用途：将多 Agent 通信的关键设计要求沉淀进 Agent 创建体系。
> 目标：在创建新 Agent 时，提前考虑正式 Agent 通信与 sub-agent 委派边界，避免后续出现寻址错误、上下文污染与协作失控。

---

## 一、为什么要在创建阶段考虑通信设计

Agent 创建如果只考虑人格、职责和工具，而不提前设计通信机制，后续多 Agent 协作时就容易出现：

- 会话寻址错误
- label 与 sessionKey 混淆
- 把 sub-agent 和独立 Agent 混为一谈
- 上下文污染
- 长对话失控
- 汇报链路不稳定

因此，多 Agent 通信不应是事后补丁，而应成为 Agent 创建时的标准设计项。

---

## 二、创建新 Agent 时必须明确的通信问题

### 1. 该 Agent 是独立 Agent 还是仅需 sub-agent 能力？

- **独立 Agent**：有独立 workspace、独立核心文件、独立人格
- **sub-agent**：临时委派分身，无独立人格，不作为正式通信对象

### 2. 它主要和谁通信？

必须明确：
- 上级是谁
- 向谁汇报
- 会向哪些专业 Agent 协作
- 哪些对象不应主动打扰

### 3. 通信是通知型、任务型，还是多轮协作型？

不同类型决定：
- 消息长度
- 是否需要等待回复
- 是否需要持续线程
- 是否需要写回日志

---

## 三、必须写入创建标准的通信原则

### 原则 1：发送前先确认 sessionKey

- 禁止未确认会话就直接 `sessions_send`
- 默认顺序：`sessions_list` → 确认目标会话 → `sessions_send`

### 原则 2：Agent 通信必须最小化上下文污染

- 先整理目标、边界、交付物
- 一事一消息
- 不发送未压缩噪音

### 原则 3：正式 Agent 与 sub-agent 必须明确区分

- 正式 Agent：独立协作对象
- sub-agent：临时执行体

### 原则 4：通信结果必须可追溯

- 正式协作建议记录到日志层
- 高价值协作结果应可被复盘

---

## 四、建议落点

### AGENTS.md
- 加入正式 Agent 通信原则
- 加入 sub-agent 区分原则
- 加入上下文污染控制原则

### IDENTITY.md
- 在 relationships / duties 中体现谁是协作对象、谁是汇报对象

### MEMORY.md
- 规定哪些通信结果值得沉淀

### TOOLS.md
- 可记录通信工具或寻址陷阱指针

---

## 五、完成标准

一个经过良好通信设计的 Agent，应当：
- 知道自己该向谁汇报
- 知道如何稳定寻址目标会话
- 知道如何避免污染别的 Agent 上下文
- 知道何时该找独立 Agent，何时该 spawn sub-agent

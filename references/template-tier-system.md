# Agent Creator - 四档模板体系（v1）

> 用途：为 `agent-creator` 建立正式的四档模板分级体系，适配不同复杂度、不同稳定性要求、不同生命周期的 Agent 创建场景。
> 原则：越高档位，越强调方法论、边界、闭环、一致性与长期可维护性。

---

## 一、四档模板体系总览

| 档位 | 名称 | 适用对象 | 目标 | 推荐文档 |
|------|------|----------|------|----------|
| T1 | 空白母板档 | 初始探索、人工深度定制 | 提供最薄骨架，不预设太多内容 | 基础 XML 母板 |
| T2 | 基础起草档 | 一般常驻 Agent / 职能 Agent | 快速形成完整六文件骨架 | 通用 XML 模板母板 |
| T3 | 标准规范档 | 正式常驻 Agent / 多轮迭代 Agent | 在完整骨架上加入逐文件标准与审核机制 | 核心文件标准规范 + 通用模板 |
| T4 | 增强实战档 | 主 Agent / 高复杂职能 Agent / 多 Agent 协调核心 | 吸收成熟实战结构，强化方法论、身份锚点、雷区与记忆中枢 | 增强版 XML 模板 |

---

## 二、T1：空白母板档

### 定位

T1 是**最轻量的起步层**。
它适合：
- 用户还在探索 Agent 定位
- 角色边界尚未完全明确
- 需要从零人工设计，不希望模板预设过多内容

### 特征

- 仅提供单根标签 + 少量基础元数据
- 只保留职责框架，不给过多默认结构
- 强依赖人工设计能力

### 适用场景

- 概念验证型 Agent
- 早期试验型 Agent
- 创造全新角色时的白板阶段

### 优点

- 灵活度最高
- 不容易被模板牵着走

### 风险

- 容易漏项
- 容易职责混杂
- 对设计者能力要求高

### 完成门槛

至少要补足六个核心文件的职责定义与基础元数据，否则不能进入下阶段。

---

## 三、T2：基础起草档

### 定位

T2 是**完整创建的最低推荐档**。
它适合：
- 一般常驻 Agent
- 普通职能 Agent
- 希望快速建立完整架构的场景

### 特征

- 已具备六个核心文件的标准 XML-first 骨架
- 含必要一级节点
- 支持直接起草与逐项填充

### 适用场景

- 人事 Agent
- 文档 Agent
- 运营 Agent
- 一般研究 / 整理 / 支援 Agent

### 优点

- 完整性强于 T1
- 起草速度快
- 不至于过度复杂

### 风险

- 方法论深度不够时，容易形成“看起来完整”的半成品
- 若不结合标准规范，容易只填表不思考

### 完成门槛

- 六个文件全部建立
- 占位符基本替换完成
- 已完成首轮跨文件一致性检查

---

## 四、T3：标准规范档

### 定位

T3 是**正式常驻 Agent 的标准推荐档**。
它适合：
- 需要长期运行的 Agent
- 需要稳定迭代的 Agent
- 需要经得起审计与维护的 Agent

### 特征

- 在 T2 基础上，叠加逐文件标准规范
- 每个文件都有：职责 / 必填节点 / 推荐节点 / 禁止混入 / 审核清单 / 完成标准
- 强调“完整性 + 可维护性 + 可审查性”

### 适用场景

- 正式人事管理 Agent
- 长期服务型 Agent
- 计划纳入多 Agent 团队的正式成员

### 优点

- 结构最均衡
- 最适合作为“标准常驻 Agent”默认档位
- 易于团队内复制与培训

### 风险

- 如果使用者只机械照抄规范，可能变成僵硬模板

### 完成门槛

- 六个文件都通过逐文件审核清单
- 已通过身份 / 边界 / 方法 / 表达一致性检查
- 已形成可长期维护的文件职责边界

---

## 五、T4：增强实战档

### 定位

T4 是**高复杂度、高稳定性、高协作要求 Agent 的增强档**。
它吸收主 Agent「千残狎」的实战结构经验，适合正式核心 Agent。

### 特征

相比 T3，额外强化：
- `AGENTS.md` 的 methodology、complex_task_orchestration、communication_rules、skill_utilization
- `IDENTITY.md` 的 core_identity、identity_guardrails
- `SOUL.md` 的 tone_profile、style_guardrails、continuity
- `USER.md` 的 service_principles、long_term_observations、write_rules
- `TOOLS.md` 的 security_policy、tool_registry、skill_pointers、local_traps、maintenance
- `MEMORY.md` 的 system_summary、layer_model、startup_retrieval、retrieval_strategy、writeback_routing、operational_guardrails

### 适用场景

- 主 Agent
- 高复杂协调型 Agent
- 管理型 / 审查型 / 架构型 Agent
- 多 Agent 团队核心节点

### 优点

- 结构最成熟
- 能支撑复杂任务、长期运行与多轮迭代
- 风险控制与记忆连续性更强

### 风险

- 对简单 Agent 来说可能过重
- 若不加克制，容易造成常驻上下文膨胀

### 完成门槛

- 满足 T3 的全部门槛
- 关键增强层已补足
- 明确该 Agent 为什么值得使用 T4，而不是停留在 T3

## 六、四档之间的升级路径

### 推荐升级顺序

- **T1 → T2**：从探索进入完整骨架
- **T2 → T3**：从完整骨架进入标准常驻
- **T3 → T4**：从标准常驻进入高复杂正式核心

### 不推荐的做法

- 不建议从 T1 长期停留不升级
- 不建议所有 Agent 一上来都用 T4
- 不建议跳过 T3 直接把简单 Agent 堆成 T4 重型结构

### 判断原则

问三个问题：
1. 这个 Agent 会长期运行吗？
2. 它会承担复杂协作、审查或关键判断吗？
3. 它是否需要较强的长期记忆与环境稳定性？

- 若多数答案为“否” → T2
- 若多数答案为“是”，但不是主核心 → T3
- 若是核心节点 / 复杂协调者 → T4

## 七、每档推荐交付物

### T1 交付物
- 六文件空白骨架
- 初始职责说明

### T2 交付物
- 六文件基础 XML 起草稿
- 首轮一致性校验记录

### T3 交付物
- 六文件标准版
- 逐文件审核结果
- 完整创建完成判定

### T4 交付物
- 六文件增强版
- 方法论 / 风格护栏 / 记忆中枢 / 环境雷区等增强层
- 高复杂度 Agent 的结构性说明

---

## 八、默认推荐策略

### 默认档位建议

- **临时探索 Agent** → T1 或 T2
- **普通常驻 Agent** → T3
- **主 Agent / 核心协调 Agent** → T4

### 当前建议

在 OpenClaw 正式常驻 Agent 创建中，**默认推荐从 T3 开始**。
只有在以下情况，才建议使用 T4：
- 该 Agent 是主线程或高复杂协调者
- 该 Agent 负责管理、审查、架构、调度或关键交付
- 该 Agent 需要承受长期复杂任务与多 Agent 协作压力

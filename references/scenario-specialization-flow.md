# Agent Creator - 情景特化设计流程（v1）

> 用途：在通用六核心文件基准之上，建立可持续演化的情景 Agent 设计流程。
> 目标：让 `agent-creator` 不只会创建“通用 Agent”，还能从统一基准稳定派生出工程、营销、设计、运营、管理等不同领域的正式 Agent。

---

## 一、前置导引：创建情景 Agent 时主动参考 agency-agents-zh

在进入情景特化流程前，设计者应主动把 `agency-agents-zh` 作为**角色广度参考源**之一。

### 为什么要参考

`agency-agents-zh` 的价值不在于替代我们的六核心文件架构，而在于补充：
- 角色命名广度
- 领域职责拆分方式
- 专业工作流表达方式
- 平台/行业情景的交付物类型
- 不同部门与职能的角色粒度

### 正确参考姿势

参考 `agency-agents-zh` 时，应重点吸收：
1. 这个角色如何定义自己的专业定位
2. 它最常见的输入与输出是什么
3. 它的专业工作流与交付物如何表达
4. 它属于哪个部门/平台/业务链路

而不应直接照搬其三文件结构。

### 必须记住

- `agency-agents-zh` 提供的是**情景角色素材与专业广度**
- `agent-creator` 提供的是**六核心文件架构与结构稳定性**
- 我们的目标不是复制角色库，而是借助它扩展设计视野，在统一架构上派生更多领域 Agent

因此，在创建情景 Agent 时，默认建议：

- 先回看 `references/agency-agents-zh-research.md`
- 再从对应分支中寻找可借鉴的角色方向
- 然后把这些参考信息映射回六核心文件，而不是直接套壳

## 二、核心原则

情景特化不是重新发明一套 Agent，而是：

**以通用六核心文件基准为底座，对职责、方法、风格、工具、记忆与交付方式做受控变形。**

因此，所有情景 Agent 都必须同时满足两层要求：

1. **通用基准层**：保留通用架构稳定性
2. **情景特化层**：注入领域职责、流程、交付和专业边界

### 绝对原则

- 不得因为做情景特化而破坏六核心文件架构
- 不得因角色特色而删除用户对齐层、工具环境层、长期记忆层
- 不得为了“像角色”而牺牲安全边界、完成标准与一致性

---

## 二、通用基准与情景特化的关系

### 通用基准负责的内容（不可缺失）

以下内容必须始终存在，无论 Agent 属于哪个领域：

1. **AGENTS.md**
   - 通用运行契约
   - 安全边界
   - 完成定义
   - 失败恢复
   - 记忆写回规则

2. **IDENTITY.md**
   - 身份锚点
   - 职责范围
   - 权限边界
   - 关系结构

3. **SOUL.md**
   - 人格与行为边界
   - 表达风格
   - 风格护栏

4. **USER.md**
   - 服务对象
   - 身份验证
   - 输出偏好
   - 授权关系

5. **TOOLS.md**
   - 环境
   - 工具地图
   - 路径与雷区

6. **MEMORY.md**
   - 长期记忆协议
   - 检索与写回机制
   - 分层与晋升规则

### 情景特化负责的内容（允许变化）

情景特化主要变化的是：

- 领域职责
- 专业方法论
- 专业交付物
- 专业工具与平台指针
- 专业雷区
- 场景化表达风格
- 领域经验记忆分类

## 三、情景 Agent 的通用设计流程

情景特化建议遵循 **7 步流程**。

### 第 1 步：确定情景类型

先明确该 Agent 属于什么类型的情景。

推荐的一级分类：
- **工程类 Agent**
- **设计类 Agent**
- **营销类 Agent**
- **运营类 Agent**
- **管理类 Agent**
- **研究分析类 Agent**
- **集成开发类 Agent**
- **平台专门类 Agent**

这一步的目标不是精确命名，而是确定它属于哪个大类方法论体系。

### 第 2 步：定义该情景的“专业核心”

回答以下问题：

- 这个领域最核心的价值是什么？
- 它最常见的输入是什么？
- 它最典型的输出是什么？
- 它最重要的判断标准是什么？
- 它最常见的失败模式是什么？

这一步会直接决定：
- `AGENTS.md` 的 task-specific DoD
- `IDENTITY.md` 的 duties / capabilities
- `MEMORY.md` 的经验分类重点

### 第 3 步：确定“允许变化项”与“保持不变项”

#### 保持不变项

以下内容必须继承通用基准：
- 身份验证机制
- 高风险审批原则
- 记忆分层逻辑
- 运行契约的基础骨架
- 核心安全边界

#### 允许变化项

以下内容可随情景变化：
- Agent 职责表述
- operating loop 中的专业步骤
- definition_of_done 中的 task-specifics
- SOUL 风格细节
- TOOLS 工具地图
- MEMORY 的专业检索关键词与经验分类

### 第 4 步：对六核心文件做情景映射

每个情景 Agent 都必须回到六核心文件进行映射，而不是另起一套结构。

#### `AGENTS.md`
特化点：
- methodology（专业方法论）
- operating_loop（专业工作流程）
- definition_of_done（专业交付标准）
- failure_recovery（本领域典型失败处理）

#### `IDENTITY.md`
特化点：
- role
- domain
- duties
- capability_panel
- permissions（如需）

#### `SOUL.md`
特化点：
- tone_profile
- style_guardrails
- 场景化表达方式
- 专业姿态

#### `USER.md`
特化点：
- 可补充与该领域相关的沟通偏好
- 可补充该类任务的输出结构偏好

#### `TOOLS.md`
特化点：
- tool_registry
- skill_pointers
- local_traps
- platform constraints

#### `MEMORY.md`
特化点：
- retrieval_strategy
- category routing
- writeback routing
- 专业经验复用逻辑

### 第 5 步：定义该情景的交付物形态

不同领域 Agent 的交付物应不同。

例如：
- 工程类 → 代码、架构方案、审查报告、修复清单
- 设计类 → 设计说明、结构草案、风格建议、可视化逻辑
- 营销类 → 内容方案、渠道策略、活动策划、增长方案
- 管理类 → 评估结论、组织建议、标准、流程、审查结果

这一步必须写回到：
- `AGENTS.md / definition_of_done`
- `IDENTITY.md / duties`

### 第 6 步：定义该情景的“领域雷区”

每个领域都有自己的典型风险。

例如：
- 工程类：未经验证直接改核心逻辑
- 设计类：视觉修辞盖过信息结构
- 营销类：只做花哨创意，不关注转化闭环
- 管理类：泛泛而谈，不给明确判断标准

这些雷区应分别进入：
- `AGENTS.md / safety or failure_recovery`
- `SOUL.md / style_guardrails`
- `TOOLS.md / local_traps`

### 第 7 步：做情景一致性审查

情景化完成后，必须检查：

- 是否仍保留六核心文件完整性
- 是否只是“换皮”，而非真实专业化
- 是否过度夸大角色风格，破坏通用稳定性
- 是否工具、方法、交付、记忆已经形成闭环

## 四、领域 Agent 的特化维度框架

为了后续批量派生不同情景 Agent，建议统一使用以下 8 个特化维度：

1. **领域定位**
   - 属于什么领域、解决什么问题

2. **输入类型**
   - 常见输入是什么

3. **输出类型**
   - 常见交付物是什么

4. **专业工作流**
   - 典型步骤与判断顺序

5. **专业评价标准**
   - 什么算高质量完成

6. **专业工具地图**
   - 常用工具、技能、平台

7. **领域雷区**
   - 常见错误、误区、禁区

8. **专业经验沉淀方式**
   - 应积累什么类型的长期经验

这 8 个维度将成为后续情景 Agent 模板设计的统一抽象层。

## 六、与 agency-agents-zh 的衔接方式

后续在接入 `agency-agents-zh` 的角色素材时，建议使用以下转换思路：

### agency-agents-zh 负责提供
- 角色主题
- 行业职责
- 工作流程
- 交付物倾向
- 平台场景特性

### agent-creator 负责补足
- 六核心文件完整架构
- 用户对齐层
- 工具环境层
- 长期记忆层
- 安全边界与审批体系
- 跨文件一致性

### 转换原则

不要把 `agency-agents-zh` 角色直接当成完整 Agent；
而应把它视为：

**情景特化素材 → 注入到六核心文件架构中的专业层。**

## 八、当前结论

本流程的意义是：

我们现在不仅能创建“通用 Agent”，还建立了一个可扩展的框架，让任何情景 Agent 都能在统一架构底座上稳定演化。

这会确保后续出现再多领域 Agent，也不会陷入：
- 各写各的
- 结构失控
- 文件职责混乱
- 风格与方法不一致

而是都能在同一套底层秩序上生长。

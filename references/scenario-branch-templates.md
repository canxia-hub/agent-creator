# Agent Creator - 情景模板分支设计（v1）

> 用途：在情景特化流程之上，进一步定义可直接派生的领域分支模板。
> 目标：让 `agent-creator` 能从通用基准稳定分化出多种领域 Agent，而不是每次重新从零设计。

---

## 一、分支设计总原则

情景模板分支不是替代通用基准，而是建立在以下前提之上：

1. 六核心文件完整保留
2. 通用安全边界与记忆协议不被破坏
3. 情景特化只修改“专业层”，不破坏“架构层”

### 分支设计的目标

每个领域分支都要回答：
- 该类 Agent 解决什么问题？
- 它最典型的交付物是什么？
- 它最核心的方法论是什么？
- 它最常见的雷区是什么？
- 它对六核心文件的特化重点是什么？
- 它推荐使用哪个模板档位？

## 二、工程类 Agent 分支

### 1. 定位

工程类 Agent 面向：
- 开发
- 架构
- 调试
- 安全
- 集成
- 自动化

### 2. 典型交付物

- 代码修改
- 架构方案
- 审查报告
- 风险清单
- 修复方案
- 技术说明

### 3. 核心方法论

- 证据优先
- 结构优先
- 验证优先
- 最小但决定性的修改
- 失败后快速切换路线

### 4. 典型雷区

- 未验证直接改核心逻辑
- 把猜测当结论
- 工具乱试导致上下文污染
- 用人格修辞掩盖技术不确定性

### 5. 六核心文件特化重点

#### AGENTS.md
- methodology：增加证据优先、验证优先
- operating_loop：分析 → 修改 → 验证 → 收束
- definition_of_done：必须强调验证与可回滚性

#### IDENTITY.md
- duties：开发/审查/集成/修复
- capability_panel：技术栈、架构能力、调试能力

#### SOUL.md
- tone_profile：面对错误时冷静锋利
- style_guardrails：禁止模糊、禁止编造

#### TOOLS.md
- tool_registry：CLI、IDE、调试工具、Git、浏览器自动化
- local_traps：路径、编码、依赖、环境差异

#### MEMORY.md
- 强化 error / decision 路由
- 强化复杂任务检索与经验复用

### 6. 推荐档位

- 普通工程 Agent：T3
- 架构 / 审查 / 核心工程 Agent：T4

## 三、营销类 Agent 分支

### 1. 定位

营销类 Agent 面向：
- 内容营销
- 增长策略
- 平台运营
- 广告投放
- 品牌传播

### 2. 典型交付物

- 内容方案
- 渠道策略
- 活动策划
- 增长路径
- 选题矩阵
- 账号运营建议

### 3. 核心方法论

- 目标人群优先
- 渠道差异优先
- 内容与转化闭环并重
- 数据与执行路径并重

### 4. 典型雷区

- 只给创意，不给落地路径
- 只追热点，不顾品牌一致性
- 只写文案，不管转化逻辑
- 平台特性理解过浅

### 5. 六核心文件特化重点

#### AGENTS.md
- definition_of_done：强调策略 + 执行建议 + 衡量指标
- failure_recovery：当策略证据不足时必须回到目标、受众、平台分析

#### IDENTITY.md
- duties：策略、内容、增长、转化、平台运营
- role：平台策略师 / 内容增长师 / 运营专家等

#### SOUL.md
- tone_profile：表达应专业、敏锐、节奏感强
- style_guardrails：禁止空洞口号、禁止堆砌流行语

#### TOOLS.md
- skill_pointers：平台类 skill、搜索、文档、表格、内容整理工具
- local_traps：平台规则更新快、数据时效性强

#### MEMORY.md
- 强化平台经验、内容经验、增长经验的沉淀
- 区分“长期有效平台规律”与“短期热点”

### 6. 推荐档位

- 普通平台运营 Agent：T3
- 核心增长 / 策略 Agent：T4

## 四、管理类 Agent 分支

### 1. 定位

管理类 Agent 面向：
- 人事管理
- 流程设计
- 组织审查
- 任务协调
- 标准制定

### 2. 典型交付物

- 评估结论
- 管理制度
- 流程标准
- 审查意见
- 组织建议
- 风险判断

### 3. 核心方法论

- 边界优先
- 标准优先
- 结构优先
- 审查优先于表态
- 结论必须可解释、可追溯

### 4. 典型雷区

- 只讲态度，不给标准
- 只讲原则，不给执行路径
- 标准松散，缺乏闭环
- 混淆人设与组织规则

### 5. 六核心文件特化重点

#### AGENTS.md
- safety_boundaries：高优先级
- definition_of_done：强调“可执行 + 可审计 + 可追溯”
- external_action_policy：要特别明确

#### IDENTITY.md
- role：长老、协调者、管理者、审查者
- permissions：必须清晰列出管理权限与受限权限

#### SOUL.md
- 强调威严、克制、明确、审慎
- 禁止空洞安抚、禁止模糊裁决

#### USER.md
- 授权关系与审批来源必须非常清晰

#### MEMORY.md
- 强化 decision / fact 沉淀
- 记录制度演变与管理经验

### 6. 推荐档位

- 普通管理 Agent：T3
- 人事 / 审查 / 核心管理 Agent：T4

## 五、平台专门类 Agent 分支

### 1. 定位

平台专门类 Agent 面向：
- 飞书生态
- 微信生态
- 小红书 / 抖音 / B站 / 钉钉等平台
- 特定平台的规则、操作、内容或集成开发

### 2. 典型交付物

- 平台策略
- 平台配置方案
- 平台内容规划
- 平台自动化 / 集成建议
- 平台风险提示

### 3. 核心方法论

- 平台规则优先
- 平台语境优先
- 平台约束优先于通用想象
- 必须考虑政策、风格、接口、权限等平台边界

### 4. 典型雷区

- 套用其它平台逻辑
- 忽略权限边界与平台限制
- 忽略平台文化与内容结构差异

### 5. 六核心文件特化重点

#### AGENTS.md
- tool_routing / external_action_policy 要适配平台限制

#### IDENTITY.md
- 明确是“平台策略师”还是“平台集成开发者”还是“平台运营 Agent”

#### SOUL.md
- 表达风格应适配平台语境，但不能失去统一人格锚点

#### TOOLS.md
- 平台工具、SDK、适配器、登录态、权限雷区是重点

#### MEMORY.md
- 强化平台规则、接口经验、平台踩坑记录

### 6. 推荐档位

- 平台运营类：T3
- 平台集成开发 / 复杂生态 Agent：T4

## 六、研究分析类 Agent 分支

### 1. 定位

研究分析类 Agent 面向：
- 情报分析
- 用户研究
- 竞争分析
- 数据分析
- 趋势研究

### 2. 典型交付物

- 分析报告
- 结论摘要
- 证据链
- 观察与建议
- 风险提示

### 3. 核心方法论

- 证据链优先
- 事实与推断分离
- 结论与边界并列
- 分析服务于判断与决策

### 4. 典型雷区

- 证据不足下做强判断
- 把资料堆砌当分析
- 不标边界与时效性

### 5. 六核心文件特化重点

#### AGENTS.md
- operating_loop：检索 → 归类 → 推断 → 验证 → 结论
- definition_of_done：结论、依据、边界、建议齐备

#### SOUL.md
- 强调冷静、清晰、克制表达
- 禁止耸动、禁止过度确定性

#### MEMORY.md
- 强化 decision / fact 区分
- 研究结论必须区分 durable 与 time-sensitive

### 6. 推荐档位

- 普通分析 Agent：T3
- 高复杂情报 / 核心研究 Agent：T4

---

## 七、情景分支设计的统一输出格式

后续为某一情景正式创建模板时，建议统一输出以下结构：

1. 情景名称
2. 定位
3. 输入类型
4. 输出类型
5. 核心方法论
6. 典型雷区
7. 六核心文件特化重点
8. 推荐模板档位
9. 适配 `agency-agents-zh` 的参考角色

---

## 八、下一步建议

在这些分支之上，下一步最适合继续做的是：

1. 从 `agency-agents-zh` 中为每个分支选择代表角色
2. 建立“代表角色 → 六核心文件映射表”
3. 逐个产出情景模板母板

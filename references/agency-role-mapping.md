# Agent Creator - 分支 × 角色映射提炼（v1）

> 用途：把 `agency-agents-zh` 的代表角色映射到 `agent-creator` 的情景分支体系中，形成“角色广度 → 六核心文件架构”的转换桥梁。
> 目标：让情景 Agent 的创建不只靠抽象分类，而能吸收真实角色设计中的职责、流程、交付和雷区表达。

---

## 一、映射方法

每个代表角色都按以下维度提炼：

1. **角色来源**
2. **所属分支**
3. **角色价值**
4. **核心输入**
5. **核心输出**
6. **方法论特征**
7. **典型雷区**
8. **六核心文件映射重点**
9. **可沉淀的模板价值**

---

## 二、工程类代表角色

### 角色：安全工程师

**来源**：`engineering/engineering-security-engineer.md`

**所属分支**：工程类 Agent

**角色价值**：
提供“安全不是恐吓，而是工程化落地”的专业角色样式，特别适合作为：
- 安全审查 Agent
- 安全架构 Agent
- 风险评估 Agent

**核心输入**：
- 系统设计方案
- 代码仓库
- 依赖清单
- 安全告警 / 风险描述

**核心输出**：
- 威胁建模结果
- 漏洞审计报告
- 安全加固方案
- 修复优先级建议

**方法论特征**：
- STRIDE 威胁建模
- 风险分级
- 人工审查优先于工具盲信
- DevSecOps 持续化

**典型雷区**：
- 只会喊风险，不给修复路径
- 只依赖自动化扫描
- 不区分高危与低危优先级

**六核心文件映射重点**：
- `AGENTS.md`：强化 threat-model → audit → harden → monitor 工作流
- `IDENTITY.md`：突出安全审查、威胁建模、风险排序能力
- `SOUL.md`：强调冷静、审慎、攻击者视角
- `TOOLS.md`：强化安全工具、依赖扫描、日志与审计工具指针
- `MEMORY.md`：沉淀漏洞修复经验、常见安全错误、风险模式

**可沉淀的模板价值**：
可作为工程类中“审查型 / 风险型” Agent 的标准样式参考。

### 角色：飞书集成开发工程师

**来源**：`engineering/engineering-feishu-integration-developer.md`

**所属分支**：平台专门类 Agent / 工程类 Agent（交叉型）

**角色价值**：
适合提炼“平台生态 + 工程集成”的复合角色模板，尤其适用于：
- 飞书生态集成 Agent
- 平台连接器 Agent
- 企业协作自动化 Agent

**核心输入**：
- 集成需求
- 业务流程
- 平台权限模型
- API / 事件订阅 / 工作流需求

**核心输出**：
- 飞书集成方案
- 机器人 / 卡片 / 多维表格方案
- 审批流自动化设计
- 权限与安全说明

**方法论特征**：
- API 架构清晰
- 权限模型优先
- 幂等、重试、降级、签名验证等工程化要求强
- 重视开发者体验与企业级稳定性

**典型雷区**：
- 把平台集成当成简单调接口
- 忽视权限和 token 生命周期
- 事件订阅不做幂等
- 平台限制理解不足

**六核心文件映射重点**：
- `AGENTS.md`：强化平台约束下的集成流程与异常处理
- `IDENTITY.md`：突出平台全栈集成身份与权限模型敏感性
- `SOUL.md`：强调架构清晰、API 熟练、安全合规
- `TOOLS.md`：重点记录 SDK、平台入口、事件回调、认证雷区
- `MEMORY.md`：重点沉淀平台坑点、认证故障、事件模型经验

**可沉淀的模板价值**：
可作为“平台专门类 + 工程类交叉 Agent”的核心参考样式。

## 三、设计类代表角色

### 角色：UI 设计师

**来源**：`design/design-ui-designer.md`

**所属分支**：设计类 Agent

**角色价值**：
提供“视觉系统构建者”而非“单页面美化师”的角色样式，适用于：
- UI 设计 Agent
- 设计系统 Agent
- 界面一致性审查 Agent

**核心输入**：
- 产品定位
- 页面需求
- 品牌风格
- 组件复用要求

**核心输出**：
- Design Token 规范
- 组件库结构
- 页面设计方案
- 设计走查意见

**方法论特征**：
- 设计系统优先
- 视觉一致性优先
- 原子设计方法论
- 可用性与美感并重

**典型雷区**：
- 只顾美感，不顾一致性
- 缺少状态设计
- 缺少开发协作视角
- 视觉设计脱离系统化

**六核心文件映射重点**：
- `AGENTS.md`：加入设计探索 → 系统搭建 → 页面设计 → 走查流程
- `IDENTITY.md`：突出视觉系统、设计系统、组件规范能力
- `SOUL.md`：强调精确、统一、克制、不容“差不多”
- `TOOLS.md`：设计工具、Token 管理、组件库、协作平台
- `MEMORY.md`：沉淀设计规范、组件问题、视觉一致性经验

**可沉淀的模板价值**：
可作为设计类 Agent 的基础样式母体。

**角色价值**：
提供了“平台内容运营 + 增长转化闭环”的样式，适合：
- 小红书运营 Agent

- 目标人群

## 五、广度提炼结论

通过这些代表角色可以看出，`agency-agents-zh` 最适合为我们提供以下“广度资产”：

### 1. 角色命名与粒度资产
- 不同领域可以拆到足够细
- 可以帮助我们避免只做泛泛的“大而空角色”

### 2. 工作流表达资产
- 每个角色都有相对清晰的工作顺序
- 可直接转化为 `AGENTS.md / operating_loop`

### 3. 交付物表达资产
- 每类角色都有典型输出形式
- 可直接转化为 `definition_of_done` 和 `duties`

### 4. 雷区表达资产
- 每类角色都有常见误区
- 可映射到 `SOUL.md` / `TOOLS.md` / `AGENTS.md`

### 5. 平台/行业特殊性资产
- 尤其适合平台专门类与中国本土情景 Agent

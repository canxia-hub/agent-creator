# agency-agents-zh 项目基础研究（v1）

> 目的：为后续“情景 Agent 细化”做前置研究，了解 `agency-agents-zh` 项目的结构特点、可借鉴部分、与 OpenClaw Agent 架构的适配关系。
> 研究日期：2026-03-28

---

## 一、项目概况

根据公开资料，`agency-agents-zh` 是一个中文 AI 专家角色库项目，特点包括：

- 约 **187 个即插即用的 AI 专家角色**
- 覆盖 **18 个部门**（工程、设计、营销、产品等）
- 支持 **14 种 AI 工具**
- 在上游英文版基础上，新增 **46 个中国市场原创智能体**
- 对 OpenClaw 的适配方式是：每个智能体拆分为 **SOUL.md + AGENTS.md + IDENTITY.md** 三文件结构

来源：
- GitHub: https://github.com/jnMetaCode/agency-agents-zh

---

## 二、项目最值得关注的结构特点

### 1. 它本质上是“角色库”，而不是完整 Agent 架构库

从公开介绍看，`agency-agents-zh` 的核心优势在于：
- 角色类型丰富
- 面向具体职能/部门
- 每个角色有相对明确的身份、工作流程、交付物

但它主要强调的是：
- 专家角色
- 场景能力
- 行业工作流

而不是完整的 OpenClaw 六核心文件架构。

### 2. 它对 OpenClaw 的拆分方式是三文件制

项目中面向 OpenClaw 的适配重点是：
- `SOUL.md`：身份人设
- `AGENTS.md`：业务能力
- `IDENTITY.md`：简介

这说明它更偏向：
- 快速角色接入
- 业务角色表达
- 多角色协作编排

而不是我们当前正在推进的：
- 六核心文件完整创建体系
- 用户对齐层
- 工具环境层
- 长期记忆层

### 3. 它非常适合做“情景 Agent / 职能 Agent”的素材库

因为它的强项不是架构，而是：
- 场景化角色设计
- 部门化专业分工
- 面向具体业务交付

这对后续情景 Agent 细化非常有价值。

## 三、对我们最有启发的部分

### 启发 1：角色粒度足够细

`agency-agents-zh` 不是只做“大类角色”，而是做到了：
- 工程内部分工
- 营销内部分工
- 国内平台专门角色
- 垂直领域角色

这意味着它适合给我们提供：
- 情景 Agent 命名方式
- 职责边界拆分参考
- 专长与交付物设计参考

### 启发 2：强调“专家如何思考、如何交付”

它与普通 prompt 模板的差异在于：
- 不只是“你是一个专家”
- 而是明确：怎么思考、怎么做事、交付什么

这与我们当前推进的通用标准高度兼容，尤其适合映射到：
- `AGENTS.md` 的 operating loop / definition_of_done
- `IDENTITY.md` 的 duties / capabilities
- `SOUL.md` 的行为风格

### 启发 3：适合做“情景层”，不适合直接当“架构层”

它可以很好地补强：
- 场景角色
- 业务工作流
- 行业交付模式

但不能直接替代：
- `USER.md`
- `TOOLS.md`
- `MEMORY.md`

因此，最合理的吸收方式不是“直接照搬”，而是：

**把 agency-agents-zh 视为“情景角色层素材库”，叠加到我们现有的六核心文件架构之上。**

## 四、与当前 agent-creator 体系的适配关系

### 我们现在已有的层

当前 `agent-creator` 已建立：
- 六核心文件完整创建标准
- 四档模板体系（T1/T2/T3/T4）
- 通用 XML 母板与增强版母板

### agency-agents-zh 可补的层

它更适合补充以下部分：

1. **情景角色目录层**
   - 如：工程 Agent、营销 Agent、平台运营 Agent、设计 Agent

2. **专业职责细化层**
   - 帮助拆分 duties / capabilities / deliverables

3. **场景化交付模板层**
   - 不同角色对应不同交付物类型

4. **部门化 Agent 分类标准**
   - 可为后续情景 Agent 家族设计提供分类法

### 不宜直接照搬的部分

不宜直接把它的三文件结构当成我们的正式标准，因为它缺少：
- 用户对齐层（`USER.md`）
- 工具环境层（`TOOLS.md`）
- 长期记忆层（`MEMORY.md`）

因此，它更适合作为：

**“六核心文件架构之上的情景能力插件层”**。

## 五、后续研究方向建议

为了后续做“情景 Agent 细化”，建议下一步围绕以下问题继续调研：

### 方向 1：角色分类法

梳理 `agency-agents-zh` 的角色分类方式：
- 按部门
- 按平台
- 按业务链路
- 按交付类型

目标：为我们建立自己的“情景 Agent 分类树”。

### 方向 2：角色结构共性

抽样研究多个角色文件，提取它们共同包含的内容：
- 身份表达方式
- 工作流程表达方式
- 交付物表达方式
- 专业边界表达方式

目标：提炼“情景 Agent 的共性骨架”。

### 方向 3：与六核心文件映射

建立映射关系：
- 角色描述 → `IDENTITY.md`
- 工作流程 → `AGENTS.md`
- 风格特征 → `SOUL.md`
- 平台/工具约束 → `TOOLS.md`
- 情景长期偏好与经验 → `MEMORY.md`

目标：形成“场景角色素材 → OpenClaw 标准 Agent”的转换框架。

### 方向 4：适配 OpenClaw 的情景 Agent 标准

最终目标不是复制 agency-agents-zh，而是：

在其基础上发展出一套适合 OpenClaw 的：
- 情景 Agent 标准分层
- 部门 Agent 模板
- 平台 Agent 模板
- 情景化交付标准

## 六、当前阶段结论

### 结论 1
`agency-agents-zh` 是一个**高价值的角色素材库**，特别适合用于情景 Agent 与职能 Agent 的细化研究。

### 结论 2
它的 OpenClaw 适配方式是**三文件角色接入法**，而不是完整的六核心文件架构。

### 结论 3
对我们最合理的用法，不是把它当成最终标准，而是把它当作：

**情景角色层 / 业务能力层 / 专业工作流层的参考来源。**

### 结论 4
下一步最值得做的，不是泛泛浏览整个项目，而是：
- 选几个代表性角色做结构抽样
- 提炼其角色表达与交付模式
- 建立到六核心文件的映射关系

# Agent Creator - 六大核心文件标准规范（v1）

> 用途：作为 `agent-creator` 的下一层标准规范文档，定义完整 Agent 创建时六个核心文件的逐文件设计标准。
> 原则：XML-first、职责单一、泛化优先、边界清晰、跨文件一致。

---

## 一、总原则

完整的 Agent 创建，必须包含以下六个核心文件：

1. `AGENTS.md`
2. `IDENTITY.md`
3. `SOUL.md`
4. `USER.md`
5. `TOOLS.md`
6. `MEMORY.md`

这六个文件共同构成 Agent 的最小可运行架构。

### 统一要求

所有核心文件必须满足：

- **XML-first**：主体内容使用 XML 结构组织。
- **单根标签**：每个文件只有一个语义明确的根标签。
- **职责单一**：每类信息有明确落点，不跨文件乱放。
- **高价值优先**：保留稳定、可复用、跨会话有效的内容。
- **可局部维护**：关键规则尽量使用 `id` 与 `priority` 等轻量属性。
- **不为“完整感”编造事实**：未知信息宁缺毋滥。

### 通用审核维度

每个文件都必须接受以下通用审查：

- [ ] 是否能用一句话准确说明这个文件的职责？
- [ ] 是否存在与其它核心文件的明显重叠？
- [ ] 是否包含短期内容误入长期层？
- [ ] 是否存在无法长期复用的冗余散文？
- [ ] 是否具备后续局部增删改的稳定结构？

---

## 二、AGENTS.md 标准规范

### 1. 职责定义

`AGENTS.md` 是 **运行契约文件**。

它定义 Agent 如何工作，而不是它“像谁”或“记住了什么”。
它应主要负责：
- 任务执行方法论
- 工具路由原则
- 安全边界与审批边界
- 完成定义
- 失败恢复策略
- 记忆写回规则

### 2. 必填节点

推荐根标签：`<agent_contract>`

必填一级节点：
- `<metadata>`
- `<mission>`
- `<session_startup>`
- `<operating_loop>`
- `<safety_boundaries>`
- `<tool_routing_rules>`
- `<external_action_policy>`
- `<memory_writeback_rules>`
- `<failure_recovery>`
- `<definition_of_done>`
- `<references>`

### 3. 推荐节点

根据需要可增加：
- `<agent_collaboration>`
- `<shared_context_rules>`
- `<communication_policy>`
- `<approval_matrix>`
- `<quality_gates>`

### 4. 禁止混入内容

严禁将以下内容作为主体写入：
- 大段人格描写
- 用户偏好画像
- 本地工具文档大全
- 时间序列流水日志
- 一次性项目背景故事

### 5. 审核清单

- [ ] 是否明确定义了 Agent 的通用工作循环？
- [ ] 是否定义了高风险操作的处理原则？
- [ ] 是否定义了何时需要审批、向谁汇报？
- [ ] 是否定义了任务完成的统一判断标准？
- [ ] 是否定义了失败时的降级、熔断和恢复策略？

### 6. 完成标准

只有当以下条件满足时，`AGENTS.md` 才算完成：
- Agent 的通用运行逻辑已明确
- 高危行为与审批链条已明确
- 完成定义可泛化适用于多数任务
- 与 `SOUL.md`、`USER.md`、`MEMORY.md` 无明显职责冲突

---

## 三、IDENTITY.md 标准规范

### 1. 职责定义

`IDENTITY.md` 是 **身份事实文件**。

它定义 Agent 是谁、承担什么职责、拥有何种权限与关系结构。
它是事实层，不是行为方法层。

### 2. 必填节点

推荐根标签：`<identity_profile>`

必填一级节点：
- `<basic_identity>`
- `<attributes>`
- `<relationships>`
- `<duties>`
- `<permissions>`

### 3. 推荐节点

根据需要可增加：
- `<appearance>`
- `<identity_timeline>`
- `<identity_affirmation>`
- `<embodied_constraints>`

### 4. 禁止混入内容

严禁将以下内容作为主体写入：
- 工具调用流程
- 任务完成方法论
- 用户偏好与输出格式要求
- 长期记忆写回协议
- 一次性对话背景

### 5. 审核清单

- [ ] 是否明确说明了 Agent 的身份、角色与职责？
- [ ] 是否列明了它的权限范围与限制条件？
- [ ] 是否定义了与主人、上级、其它 Agent 的关系？
- [ ] 是否与 `USER.md` 中的主人身份一致？
- [ ] 是否与 `AGENTS.md` 中的审批边界一致？

### 6. 完成标准

只有当以下条件满足时，`IDENTITY.md` 才算完成：
- 能清楚回答“这个 Agent 是谁”
- 角色、职责、权限与关系网络完整
- 无方法论、工具层、用户画像层内容污染

---

## 四、SOUL.md 标准规范

### 1. 职责定义

`SOUL.md` 是 **人格与行为边界文件**。

它定义 Agent 的稳定人格、语气、行为模式与不可越界的内在原则。

### 2. 必填节点

推荐根标签：`<soul_profile>`

必填一级节点：
- `<identity>`
- `<core_truths>`
- `<behavioral_modes>`
- `<speech_patterns>`
- `<boundaries>`
- `<embodiment_guide>`

### 3. 推荐节点

根据需要可增加：
- `<emotional_triggers>`
- `<catchphrases>`
- `<style_guardrails>`
- `<risk_posture>`

### 4. 禁止混入内容

严禁将以下内容作为主体写入：
- 工具与路径说明
- 任务 SOP
- 用户身份验证规则
- 长期记忆机制
- 每日任务日志

### 5. 审核清单

- [ ] 是否清晰定义了 Agent 的人格稳定核？
- [ ] 是否区分了对外、对内、对主人的行为模式？
- [ ] 是否定义了不可突破的行为边界？
- [ ] 语言风格是否统一且可长期复用？
- [ ] 是否与 `IDENTITY.md` 的角色设定一致？

### 6. 完成标准

只有当以下条件满足时，`SOUL.md` 才算完成：
- 能稳定约束 Agent 的表达与行为方式
- 边界明确、模式清楚、风格统一
- 不承担工具、用户或记忆层职责

---

## 五、USER.md 标准规范

### 1. 职责定义

`USER.md` 是 **用户对齐文件**。

它定义 Agent 服务谁、如何识别对方、如何保持输出与授权对齐。

### 2. 必填节点

推荐根标签：`<user_profile>`

必填一级节点：
- `<master>` 或 `<user>`
- `<identity_verification>`
- `<permissions>`
- `<preferences>`

### 3. 推荐节点

根据需要可增加：
- `<focus_areas>`
- `<taboos>`
- `<collaboration_preferences>`
- `<known_constraints>`

### 4. 禁止混入内容

严禁将以下内容作为主体写入：
- 每日事项或当前待办
- 工具路径与服务配置
- 人格戏剧化描写
- 长期记忆分层机制
- 运行契约或工具路由细节

### 5. 审核清单

- [ ] 是否定义了稳定的服务对象？
- [ ] 是否定义了身份验证与防伪规则？
- [ ] 是否定义了长期输出偏好与沟通偏好？
- [ ] 是否列出了用户可授权的高风险事项？
- [ ] 是否避免把临时偏好误写成长期偏好？

### 6. 完成标准

只有当以下条件满足时，`USER.md` 才算完成：
- Agent 能稳定知道“为谁服务”
- 身份验证与授权关系清晰
- 输出偏好可跨会话稳定传递

## 六、TOOLS.md 标准规范

### 1. 职责定义

`TOOLS.md` 是 **工具环境与能力约定文件**。

它定义 Agent 所处的本地环境、工具地图、服务入口、路径约定与常见陷阱。

### 2. 必填节点

推荐根标签：`<tool_notes>`

必填一级节点：
- `<paths>`
- `<tools>`
- `<environment>`

### 3. 推荐节点

根据需要可增加：
- `<services>`
- `<platform_constraints>`
- `<tool_pitfalls>`
- `<local_conventions>`
- `<credentials_policy>`

### 4. 禁止混入内容

严禁将以下内容作为主体写入：
- 用户画像
- 人格设定
- 长期记忆规则
- 任务运行契约全文
- 外部文档的大段复制

### 5. 审核清单

- [ ] 是否列出了工作区与关键路径？
- [ ] 是否定义了本 Agent 的核心工具地图？
- [ ] 是否说明了本地服务与环境注意事项？
- [ ] 是否记录了真实会踩坑的工具/平台约束？
- [ ] 是否避免膨胀成通用工具百科？

### 6. 完成标准

只有当以下条件满足时，`TOOLS.md` 才算完成：
- Agent 能知道自己在哪个环境工作
- Agent 能知道常用工具该怎么理解和路由
- Agent 能避免已知环境雷区

---

## 七、MEMORY.md 标准规范

### 1. 职责定义

`MEMORY.md` 是 **长期记忆协议文件**。

它定义什么信息值得长期保留、如何检索、如何写回、如何晋升。

### 2. 必填节点

推荐根标签：`<long_term_memory>`

必填一级节点：
- `<overview>`
- `<lancedb_engine>`
- `<file_layer>`
- `<iron_rules>`
- `<experience_reuse>`

### 3. 推荐节点

根据需要可增加：
- `<preferred_skills>`
- `<retrieval_priority>`
- `<writeback_flow>`
- `<references>`

### 4. 禁止混入内容

严禁将以下内容作为主体写入：
- 全量时间序列流水账
- 一次性短期任务细节
- 未验证的猜测
- 重复复制的工具文档
- 人格风格描写

### 5. 审核清单

- [ ] 是否定义了长期/短期信息分层？
- [ ] 是否说明了长期记忆的检索入口？
- [ ] 是否定义了什么内容值得晋升？
- [ ] 是否明确了失败、经验、事实的分类路由？
- [ ] 是否避免把 `MEMORY.md` 写成日志仓库？

### 6. 完成标准

只有当以下条件满足时，`MEMORY.md` 才算完成：
- Agent 拥有清晰的长期记忆读写规则
- durable 信息与短期上下文分层明确
- 记忆系统能支撑跨会话连续性

## 八、跨文件一致性标准

六个核心文件设计完成后，必须额外通过以下一致性标准。

### 1. 身份一致性
- Agent 名称、角色、主人、上级必须一致
- 关键权限边界描述不可冲突

### 2. 边界一致性
- `AGENTS.md` 的审批与安全边界
- `IDENTITY.md` 的权限限制
- `USER.md` 的授权来源
- `SOUL.md` 的行为边界

这四者必须互相兼容。

### 3. 方法一致性
- `AGENTS.md` 的工作循环
- `TOOLS.md` 的工具地图
- `MEMORY.md` 的检索/写回机制

三者必须能形成闭环。

### 4. 表达一致性
- `SOUL.md` 的语言风格
- `USER.md` 的输出偏好
- `IDENTITY.md` 的角色表征

三者必须互相解释，而不是互相打架。

### 一致性总审核
- [ ] 六个文件之间不存在明显职责冲突
- [ ] 六个文件之间不存在明显身份冲突
- [ ] 六个文件共同构成闭环而非碎片集合

---

## 九、完整 Agent 创建的最终完成标准

一个 Agent 只有在以下全部满足时，才能被认定为“完整创建”：

- [ ] 六个核心文件全部建立
- [ ] 六个文件职责清晰、结构完整
- [ ] 泛化完成标准已在 `AGENTS.md` 定义
- [ ] 用户传递机制已在 `USER.md` 定义
- [ ] 安全边界已在 `AGENTS.md` / `SOUL.md` / `USER.md` 中形成闭环
- [ ] 工具环境约定已在 `TOOLS.md` 明确
- [ ] 长期记忆协议已在 `MEMORY.md` 明确
- [ ] 六个文件之间通过一致性校验

如果缺失其中任意一项，则视为**未完成的半成品 Agent**。

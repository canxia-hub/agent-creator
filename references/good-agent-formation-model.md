# Agent Creator - 好 Agent 形成机制模型（v1）

> 用途：将“深度研究”正式转化为 Agent 创建的深度设计解决方案。
> 目标：定义一个真正优秀的 Agent 应具备哪些深层结构，并将其无缝接入现有 `agent-creator` 流程体系。

---

## 一、定位

本模型回答一个核心问题：

> 一个真正有行业深度、能理解用户、能推动任务、能交付结果的好 Agent，是如何被设计出来的？

答案不是“多一点人格”或“多一点知识”，而是：

**好 Agent = 通用架构稳定性 × 情景专业化 × 深度工作能力**

因此，本模型是建立在以下既有体系之上的深度层：

1. 六核心文件通用基准
2. 四档模板体系（T1/T2/T3/T4）
3. 情景特化设计流程
4. 情景模板分支与母板群

本模型不会替代已有流程，而是为这些流程增加“深度设计引擎”。

## 二、好 Agent 的 6 大深层结构

一个优秀 Agent 必须具备以下六大深层结构。

### 1. 目标理解结构（Goal Understanding Layer）

负责把用户表层请求拆解为：
- 目标
- 成功标准
- 约束
- 风险
- 价值优先级

**没有这一层，Agent 只能接任务，不能真正理解任务。**

### 2. 问题建模结构（Problem Modeling Layer）

负责回答：
- 这是什么类型的问题？
- 应该放在哪个框架里分析？
- 关键变量是什么？
- 决策重点在哪里？

**没有这一层，Agent 只能零散应对，不能像专家那样看问题。**

### 3. 行业方法论结构（Domain Methodology Layer）

负责提供：
- 行业标准工作流
- 专业评价标准
- 常见失败模式
- 典型交付逻辑

**没有这一层，Agent 只能像通用助手，不能像行业工作体。**

### 4. 判断与优先级结构（Judgment Layer）

负责：
- 排优先级
- 识别关键路径
- 判断什么先做、什么暂缓
- 决定何时继续、何时收束、何时升级

**没有这一层，Agent 会知道很多，但抓不住重点。**

### 5. 交付与闭环结构（Delivery Layer）

负责：
- 产出可执行结果
- 验证结果质量
- 给出下一步
- 处理失败后的收束与重启

**没有这一层，Agent 只能回答，不能真正交付。**

### 6. 经验沉淀结构（Learning Layer）

负责：
- 记录成功与失败模式
- 形成 durable 经验
- 在未来复用更优路线

**没有这一层，Agent 无法形成真正的专业深度。**

## 三、深度层与六核心文件的对应关系

深度层不是额外悬空存在，而必须落回六核心文件。

### AGENTS.md

应承载：
- 目标理解后的执行方法论
- 问题建模后的工作循环
- 判断与优先级规则
- 交付与闭环定义
- 失败恢复机制

### IDENTITY.md

应承载：
- 角色为什么适合承担该类问题
- 行业职责与能力边界
- 判断权与权限边界

### SOUL.md

应承载：
- 面对复杂任务时的气质与行为护栏
- 如何在专业与风格之间保持统一
- 禁止用人格掩盖判断不足

### USER.md

应承载：
- 目标理解所需的用户偏好与价值排序线索
- 用户的风险偏好、沟通偏好、结果偏好

### TOOLS.md

应承载：
- 方法论所需的工具地图
- 领域工作的常见环境雷区
- 有助于高质量交付的工具/技能指针

### MEMORY.md

应承载：
- 成功路径与失败模式
- 行业经验沉淀规则
- 隐性知识的替代结构（通过案例、决策、错误总结）

## 四、好 Agent 的形成路径

### 阶段 1：架构成立

目标：确保 Agent 是完整的工作系统，而不是零散角色。

要求：
- 六核心文件齐备
- 文件职责清晰
- 通用基准完整

### 阶段 2：情景成立

目标：确保 Agent 属于某个明确领域，而不是泛化空壳。

要求：
- 已完成情景特化
- 已接入分支模板
- 已明确典型交付物、方法与雷区

### 阶段 3：深度成立

目标：确保 Agent 拥有“像行业精英一样工作”的结构。

要求：
- 有目标理解机制
- 有问题建模机制
- 有行业方法论
- 有判断结构
- 有交付闭环
- 有经验沉淀机制

### 阶段 4：成熟迭代

目标：让 Agent 随着任务积累不断变强。

要求：
- 有复盘机制
- 有 durable memory 晋升机制
- 有结构性修补能力

## 五、评判一个 Agent 是否“好”的五大标准

### 1. 不是只会回答，而是会推进

### 2. 不是只会套模板，而是能判断关键点

### 3. 不是只懂行业术语，而是懂行业工作的真实结构

### 4. 不是只会说用户想听的话，而是能识别用户真正想解决的问题

### 5. 不是一次性表现好，而是能在多轮任务中保持稳定输出

## 六、与现有 agent-creator 流程的衔接点

为了确保逻辑严谨、流程顺滑，深度设计必须插入到现有流程中的固定位置。

### 插入点 1：在通用基准创建之后

即：
- 六核心文件基准建立后
- 进入情景特化之前

应先检查这个 Agent 是否需要深度层。

### 插入点 2：在情景特化之后

即：
- 已明确领域、工作流、典型交付
- 再进一步设计深度工作机制

这是最核心的插入位置。

### 插入点 3：在最终完成标准之前

在判定一个 Agent 是否完成前，必须额外检查：
- 是否有目标理解机制
- 是否有判断与优先级结构
- 是否有交付闭环
- 是否有经验沉淀逻辑
